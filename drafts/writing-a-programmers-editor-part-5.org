In [[/blog/2018/08/25/writing-a-programmers-editor-part-4/][Part 4]] we built a line management layer and a simple viewport renderer. But our
renderer just returns strings — it does not actually paint them on a screen. To do that,
we need to talk to the terminal.

This part covers:

- How terminals actually work (the teletype heritage).
- Switching from /cooked mode/ to /raw mode/ via termios.
- Reading keystrokes one at a time instead of line-buffered.
- Writing escape sequences to move the cursor, clear the screen, and set colours.
- Foreign function interface (FFI) in Scheme to call C library functions like =tcsetattr=.

The terminal is the editor's canvas. Understanding it deeply is essential — every screen
update, every keystroke, every colour change flows through it.

Watchout for this assay when it is published, till then.
