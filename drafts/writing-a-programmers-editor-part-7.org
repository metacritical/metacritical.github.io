In [[/blog/2018/09/08/writing-a-programmers-editor-part-6/][Part 6]] we will have wired keystrokes to commands. Now every time the user types a
character, the buffer changes and the screen must update. The naive approach — clear the
screen and redraw everything — flickers badly and wastes CPU.

This part covers:

- Why full repaints are bad: flicker, latency, and wasted I/O.
- *Dirty line tracking:* marking only the lines that changed.
- *Differential rendering:* computing the minimal set of terminal operations needed.
- Cursor positioning: moving the terminal cursor instead of redrawing.
- Emacs' redisplay engine: how it decides what to repaint and when.
- Implementing an incremental redisplay loop in Scheme.

A good redisplay engine is invisible — the screen just /feels* responsive. A bad one makes
the editor feel sluggish even on a fast machine.
